Backstage is an open platform for building developer portals. Powered by a centralized software catalog, Backstage restores order to your microservices and infrastructure and enables your product teams to ship high-quality code quickly — without compromising autonomy.

Short History

It was open-sourced during Spotify's internal Hack Week in March 2020. Backstage was accepted to CNCF on September 8, 2020 and is at the Incubating project maturity level.

Architectural Overview of Backstage

  • Developed with Typescript
  • Frontend built with React
  • API powered by Express.js

Backstage allows you to fully customize your platform. With TypeScript, you have the freedom to modify every aspect of your application. Extend its functionality effortlessly by installing plugins through yarn.

Software Catalog

The Backstage Software Catalog is a centralized system that keeps track of ownership and metadata for all the software in your ecosystem (services, websites, libraries, data pipelines, etc). The catalog is built around the concept of metadata YAML files stored together with the code, which are then harvested and visualized in Backstage.

Example Yaml

                    
apiVersion: backstage.io/v1alpha1
kind: Component
metadata:
  name: backstage
  annotations:
    github.com/project-slug: pascal-sochacki/BackstageTalk
spec:
  type: website
  lifecycle: production

EntityProvider

EntityProvider: Seamless Ingestion of Entities

The EntityProvider is an internal API in Backstage that facilitates the smooth ingestion of entities.

It supports multiple sources, including popular platforms like GitHub, GitLab, and AWS, enabling effortless integration.

EntityProvider - GitHub Integration

The EntityProvider in Backstage integrates with GitHub repositories to retrieve entity information.

It periodically scans all repositories, searching for a file named catalog-info.yaml in the default branch.

  • Enables self-service capabilities
  • Keeps entity information close to the code
  • Aligns with developer workflows

Software Templates

The Software Templates part of Backstage is a tool that can help you create Components inside Backstage.

Creating a New Service: Time-Intensive Process

  • Code generation (e.g., Spring Initializer) for project setup
  • Library installation for required dependencies
  • Git setup for version control and collaboration
  • Service configuration to meet specific requirements
  • CI pipeline setup with comprehensive scanning
  • Deployment orchestration for seamless production transition

Launching a New Service: Key Considerations

  • Alignment with company standards (Libraries, Frameworks, Language)
  • Robust security measures (Dependency updates, Security Scanning)
  • Monitoring and observability tools

apiVersion: scaffolder.backstage.io/v1beta3
kind: Template
metadata:
  name: example-nodejs-template
  title: Example Node.js Template
  description: An example template for the scaffolder that creates a simple Node.js service
spec:
  owner: user:guest
  type: service

  # These parameters are used to generate the input form in the frontend, and are
  # used to gather input data for the execution of the template.
  parameters:
    - title: Fill in some steps
      required:
        - name
      properties:
        name:
          title: Name
          type: string
          description: Unique name of the component
          ui:autofocus: true
          ui:options:
            rows: 5
    - title: Choose a location
      required:
        - repoUrl
      properties:
        repoUrl:
          title: Repository Location
          type: string
          ui:field: RepoUrlPicker
          ui:options:
            allowedHosts:
              - github.com

  # These steps are executed in the scaffolder backend, using data that we gathered
  # via the parameters above.
  steps:
    # Each step executes an action, in this case one templates files into the working directory.
    - id: fetch-base
      name: Fetch Base
      action: fetch:template
      input:
        copyWithoutTemplating:
          - '**/build.yaml'
        url: ./content
        values:
          repoOwner: ${{ (parameters.repoUrl | parseRepoUrl).owner }}
          repoName: ${{ (parameters.repoUrl | parseRepoUrl).repo }}
          name: ${{ parameters.name }}

    # This step publishes the contents of the working directory to GitHub.
    - id: publish
      name: Publish
      action: publish:github
      input:
        repoVisibility: public
        allowedHosts: ['github.com']
        description: This is ${{ parameters.name }}
        repoUrl: ${{ parameters.repoUrl }}

    # The final step is to register our new component in the catalog.
    - id: register
      name: Register
      action: catalog:register
      input:
        repoContentsUrl: ${{ steps['publish'].output.repoContentsUrl }}
        catalogInfoPath: '/catalog-info.yaml'

  # Outputs are displayed to the user after a successful execution of the template.
  output:
    links:
      - title: Repository
        url: ${{ steps['publish'].output.remoteUrl }}
      - title: Open in catalog
        icon: catalog
        entityRef: ${{ steps['register'].output.entityRef }}
